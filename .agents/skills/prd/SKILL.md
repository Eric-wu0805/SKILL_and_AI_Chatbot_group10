# /prd — 產品需求文件（PRD）生成 Skill

## 目標

根據使用者提供的專案描述，產出一份結構完整的產品需求文件（Product Requirements Document），儲存於 `docs/PRD.md`。

## 執行步驟

1. **確認專案背景**：詢問使用者關於專案的目標、目標使用者、核心功能需求。
2. **釐清範圍**：確認 MVP（最小可行產品）的功能邊界，區分「必做」和「未來再做」。
3. **撰寫 PRD**：依照下方模板，產出完整的 `docs/PRD.md` 文件。

## 輸出格式（docs/PRD.md）

請依照以下結構撰寫：

```markdown
# 產品需求文件（PRD）

## 1. 專案概述
- 專案名稱
- 專案簡介（一段話描述此產品要解決的問題）

## 2. 目標與範圍
- 專案目標（列出 2-3 個明確目標）
- 目標使用者
- 使用情境（至少 2 個）

## 3. 功能需求（Functional Requirements）

### 3.1 核心功能
| 功能名稱 | 說明 | 優先級 |
|---------|------|--------|
| ...     | ...  | P0/P1/P2 |

### 3.2 功能詳細描述
針對每個核心功能，描述：
- 使用者操作流程
- 系統回應行為
- 邊界條件與錯誤處理

## 4. 非功能需求（Non-Functional Requirements）
- 效能需求
- 安全性需求
- 可用性需求
- 相容性需求

## 5. 資料需求
- 需要儲存的資料類型
- 資料格式與欄位說明

## 6. 外部依賴
- 使用的第三方 API 或服務
- 環境需求

## 7. 驗收標準（Acceptance Criteria）
- 列出可測試的驗收條件
```

## 注意事項

- 如果使用者未提供足夠資訊，請主動提問釐清需求。
- 功能優先級使用 P0（必要）、P1（重要）、P2（期望）分類。
- 產出的文件應具備足夠細節，讓後續的架構設計和開發可以直接參考。
